# 1세트 — ① 빠르게 만들고 안정적으로 운영한다

## S1a — 강점

**빠름 — 운영 부담이 적게 설계하니 제품에 집중한다**
- 서버리스/매니지드 우선 → 운영 부담·고정비 최소화, 트래픽은 탄력 대응
- AWS Lambda + SAM에 매우 익숙 → 구현부터 배포까지 한 사이클이 짧다
- 인프라 비용을 설계 단계에서 줄이니, 같은 시간에 제품 가설을 더 많이 검증

**안정 — 그 인프라를 24/7 대규모로 운영했다 (≠ 인프라 약점)**
- 24시간 멈추면 안 되는 서비스 → **일 최고 70만 대화 무중단 운영**
- 워커 타임아웃·LLM 프로바이더 장애·메모리 폭증 등 여러 장애를 직접 겪고 대응
- 빠르게 만드는 것과 안정적으로 굴리는 것을 **한 사람이 끝까지** 책임졌다

## S1b — 뒷받침 프로젝트: 캐릭터 챗봇 API

- **한 줄**: 사용자가 AI 캐릭터와 대화·관계를 형성하는 페르소나 챗봇 (메인 제품)
- **역할**: 0→1 설계·구현·운영 전담 (2024.01–2026.05)
- **기능 5종**: 캐릭터 챗 · 속마음 엿보기 · 사진 보내기 · 페르소나 자동 생성 · 상황 이미지
- **성과**: MAU 3만 · 일 70만 대화 · 구글플레이 소셜 6위 / 앱스토어 107위 · TIPS·예창패·구글창구

## S1c — 아키텍처: 메인 시스템

```
                    [App Client]
                         │
                         ▼
        ┌──────────────────────────────────┐
        │  캐릭터 챗봇 API                   │
        │  AWS ECS(오토스케일)·FastAPI/gunicorn │
        └──────────────────────────────────┘
          │              │                │
          ▼              ▼                ▼
   ElastiCache     대화 요약+         사진 이해
   (Redis)         프롬프트 캐싱       (멀티모달)
   세션·기억            │
                       ▼
                [LLM Provider] (폴백·재시도)
```
- **비용 선형 증가 차단** — 대화 요약 + 프롬프트 캐싱으로 건당 30%↓ (일 70만 건 규모)
- **워크로드별 선택** — 동기 API는 ECS, 비동기/탄력 작업은 Lambda+SQS로 분리
- **안정성** — ECS 오토스케일로 피크 대응 + 장애 대응으로 무중단 운영

## S1d — 뒷받침 프로젝트: 선톡 전송 시스템 (서버리스)

- **문제**: "내가 말 걸 때만 응답하는 관계"로는 이탈 → **캐릭터가 먼저 말 걸어오는 경험**으로 유대·리텐션
- **결정**: 메인 서버에 붙이지 않고 **별도 Lambda(SAM)+SQS로 분리** → 복잡성 증가 차단
- **한 일**: 채팅방별 LLM 호출로 페르소나 반영 선톡 생성, SQS 비동기로 동시 호출 폭주 방지
- **디테일**: 7일간 실제 생성 결과 약 5만 건을 직접 검토하며 프롬프트 반복 개선
- **성과**: 일 8,000건 발송 안정 운영

```
 Step Function(선톡 대상 추출) → SQS(대상 채팅방) → Lambda(SAM, LLM 메시지 생성) → 기존 발송 시스템 재사용
```
> 상시 서버 없이 이벤트 기반 Lambda+SQS로 대량 처리 — "운영 부담이 적게 설계한다"(강점 ①)의 대표 사례

# 2세트 — ② 경계 없는 실행

## S2a — 강점

**시키기 애매한 일을 자발적으로 도맡는다**
- 개발에 머물지 않고 제품이 굴러가는 데 필요한 일이면 직접 메운다
- 직접 담당: QA · 공지/이벤트 웹뷰 · 이메일 CS · 콘텐츠 검열 등 운영 전반

**그래서 남들이 못 보는 진짜 병목을 안다**
- 직접 부딪힌 페인포인트를 곧장 도구·자동화로 전환 → **운영 자동화 9건**
- "직접 운영했기에 진짜 병목을 알았다" — 추측이 아니라 현장에서 나온 우선순위

## S2b — 뒷받침 프로젝트: 운영 도구 내재화 9건

- **맥락**: 전담 운영자 부재 → 본인이 운영까지 맡으며 누적된 수작업·SQL을 시스템으로 전환
- **관리자 페이지 4종**: 큐레이션·공지·테스트·운영 어드민 → 개발자 SQL 처리 → 담당자 셀프서비스
- **Slack 인터랙션 5종**: 하트 지급·계정/콘텐츠 제재·CS 라우팅·불건전 챗봇 필터 → 버튼 한 번
- **자동화 5종**: 이벤트 보상 자동 지급 · 앱 테스트(Appium) · 이미지 자동 검열 · 로그 리포트 등
- **의의**: "운영 담당자 없이도 돌아가는 운영 체계" — 모바일에서도 즉시 대응

## S2c — 아키텍처: 이벤트 기반 운영 자동화

```
 [이메일 CS] [이벤트 응모] [부적절 콘텐츠 감지]
       │          │              │
       ▼          ▼              ▼
   ┌────────────────────────────────┐
   │   이벤트 기반 Lambda            │
   └────────────────────────────────┘
       │                         │
       ▼                         ▼
  Slack 인터랙션            DB 처리
  (Block Kit·버튼)          (지급·제재·검열)

       │ 모바일에서 버튼 한 번
   [담당자 / 팀]
```

- **리스크 대응**: 이미지 노출 사고 → NSFW 자동 검열 + Slack 알림으로 재발 방지

# 3세트 — ③ 빠른 러닝커브 (AI 레버리지)

## S3a — 강점

> 특정 스택에 묶이지 않는다 — AI를 지렛대로 새 기술도 빠르게 적응해 *제품까지* 만든다.

- **프론트엔드(React) 미경험** → AI 코딩 도구로 맛보기 채팅 웹 직접 구현·배포
- **이미지 생성(Stable Diffusion) 새 도메인** → 인프라 0→1 + 서빙 엔진 직접 제작·공개
- "읽고 넘기지 않고 바로 적용" — 새 방식이 보이면 지금 프로젝트에 즉시 적용해 스킬로 굳힘
- 결론: **기술스택은 제약이 아니다. 필요한 걸 빠르게 익혀 제품으로 만든다.**

## S3b — 뒷받침 프로젝트: 이미지 생성 서비스 0→1

- **문제**: 캐릭터 이미지 만들려고 사용자가 외부(해외) 서비스로 이탈 / SaaS API 단가 압박
- **결정**: "앱 내 무료 + 빠른 생성"을 차별점으로 → 경험 없던 이미지 생성을 직접 구축
- **핵심 결정**: ① GPU 직접 계약(호출당→시간당) ② 서빙 엔진 자체 구현 ③ 번역 캐싱 ④ NSFW 자동 감지
- **성과**: 일 3,000건 · **5초 이내**(경쟁사 대비 최대 12배) · **월 24만원**(SaaS 대비 5배↓)

## S3c — 아키텍처: GPU 서빙 엔진

```
 [App] → 이미지 생성 API(FastAPI)
              │
              ▼
        한→영 번역 캐싱  (최초 1회만 번역, 이후 캐시 히트)
              │
              ▼
   ┌────────────────────────────────────────┐
   │  ComfyUI API Server (직접 구현·공개)      │
   │  · 헤드리스 + SD WebUI 호환 REST(drop-in)│
   │  · base64 메모리 처리(디스크 I/O 제거)   │
   │  · 모델 LRU 캐싱(VRAM 유지)              │
   └────────────────────────────────────────┘
              │  on 가비아 GPU (RTX A5000, 시간당 계약)
              ▼
        NSFW 자동 감지 → 결과 반환
```
- **트레이드오프 해결**: ComfyUI(빠름·API 없음) ↔ SD WebUI(API 좋음·느림) → 둘을 동시에
- **비용 구조 전환**: 호출당 단가 → 시간당 단가라 트래픽 늘수록 단위 비용 하락

## S3d — 뒷받침 프로젝트: 맛보기 채팅 웹 (React)

- **문제**: 비오가닉 유저의 가입 전환 저하 (번호 가입 거부감 · 앱 설치 마찰)
- **해결**: 가입 UX 개선 대신 **"가입 전 체험 별도 진입점"** 신설 → 웹에서 인기 캐릭터와 즉시 채팅
- **React 미경험에서 구현** → AI 코딩 도구(바이브 코딩)로 직접 구현·배포 (빠른 러닝커브의 직접 증거)
- **바이럴 루프**: 틱톡 홍보 → 웹 맛보기, 앱 내 공유 → 웹 링크 → 수신자 체험 → 앱 다운로드

```
 [틱톡/공유 링크] → 맛보기 채팅 웹(React) → S3 + CloudFront + Route 53 (서버리스 정적 호스팅)
```
> 미경험 스택(React)도 도구를 지렛대로 제품까지 직접 출시 — 정적 프론트는 서버리스로 일관

# 4세트 — ④ 시행착오라는 자산

## S4a — 강점: 깨짐 → 재발 방지

> 실패는 비용이 아니라 자산이다 — 같은 실수를 반복하지 않게 해주니까.

**완벽주의 → 최속 검증**
- 깨짐: 완벽히 다듬어 출시했더니 아무도 안 원함 / 잡은 수요는 기다리다 이탈
- 재발 방지: 오버엔지니어링 경계 + MVP 규율(좋은 아이디어도 의도적으로 버림)

**기능 집착(solution-first) → 문제부터 보기(problem-first)**
- 깨짐: 지표 하락에 데이터 진단 없이 "기능 부족"이라 단정, 기능만 추가
- 재발 방지: 문제부터 정의하고, 가장 단순한 가설부터 검증

**쉬운 것 우선(effort-first) → 임팩트 우선(impact-first)**
- 깨짐: 구현 난이도로 우선순위를 매겨 중요한 건 미루고 쉬운 것만 만듦
- 재발 방지: 난이도가 아니라 임팩트로 줄을 세운다

## S4b — 가장 비싼 교훈이 만든 '제2의 무기': 그로스해킹

> 숫자(대화량·발송량)는 쫓았지만 "왜 남고 왜 떠나는지"를 데이터로 검증하는 사이클이 빠져 있었다 — 그 공백을 정면으로 메우는 중.

- **인지**: 개발자의 가치(버그 없이 안정 운영)만으로는 스타트업에서 부족 → 제품을 *성장*시키는 역량 필요
- **방향**: 그로스해킹을 '제2의 무기'로 정하고 개발만큼 깊게 — AARRR·리텐션·PMF·Customer Centricity 내재화
- **지금 하는 것**: Statsig 도입으로 A/B 실험 인프라 구축(기능 효과를 추측 아닌 데이터로 측정)
- **메시지**: "개발 잘하는 사람"보다 **"개발도 하면서 제품이 성장하게 만드는 사람"** — 성장은 진행형

> ④는 별도 프로젝트 슬라이드 없이 회고/철학으로 마무리 (앞선 ①②③ 프로젝트들이 이미 이 교훈의 실천 결과)
