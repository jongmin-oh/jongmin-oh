## Slide 1 — Summary

**오종민 (Jongmin Oh)** · AI Engineer / Co-founder @ Reppley
> 0→1에 최적화된 AI 엔지니어 — 빠르게 만들고, 경계 없이 메우고, 같은 실패를 반복하지 않는다.

**나를 정의하는 4가지**
1. **빠르게 만들고 안정적으로 운영** — 운영 부담이 적게 인프라를 설계해 빠르게 만들고, 24/7 대규모 서비스를 무중단 운영한다
2. **경계 없는 실행** — 누구에게 시키기도 애매한 일을 자발적으로 도맡아 진짜 병목을 찾는다
3. **빠른 러닝커브** — AI를 지렛대로 어떤 기술스택도 빠르게 적응해 제품까지 만들어낸다
4. **시행착오라는 자산** — 초기 스타트업에서 몸소 깨지며, 같은 실수를 막는 법을 경험으로 안다

**증거가 되는 숫자**
- 캐릭터 챗봇 서비스 0→1 — MAU 3만 · 일 최고 70만 대화 직접 설계·구현·운영
- 운영 자동화 9건

## Slide 2 — ① 빠르게 만들고 안정적으로 운영한다

**빠름 — 운영 부담이 적게 설계하니 제품에 집중한다**
- 서버리스/매니지드 우선 설계 → 운영 부담·고정비 최소화, 트래픽은 탄력 대응
- AWS Lambda + SAM에 매우 익숙 → 구현부터 배포까지 한 사이클이 짧다
- 인프라 운영 비용을 설계 단계에서 줄이니, 같은 시간에 제품 가설을 더 많이 검증
- 0→1 증거: 캐릭터 챗봇(MAU 3만) · 이미지 생성 인프라(일 3,000건·5배↓) · 선톡(Lambda+SQS 일 8,000건)

**안정 — 그 인프라를 24/7 대규모로 운영했다 (≠ 인프라 약점)**
- 캐릭터 챗봇은 24시간 멈추면 안 되는 서비스 → **일 최고 70만 대화 무중단 운영**
- AWS ECS(오토스케일)+ElastiCache 기반으로 트래픽 급증·피크를 견디는 구조 설계·운영
- 워커 타임아웃·LLM 프로바이더 장애·메모리 폭증 등 여러 장애를 직접 겪고 대응 → 안정 운영 경험 축적
- 핵심 메시지: 빠르게 만드는 것과 안정적으로 굴리는 것을 **한 사람이 끝까지** 책임졌다

## Slide 3 — ② 경계 없는 실행

**시키기 애매한 일을 자발적으로 도맡는다**
- 개발에 머물지 않고 제품이 굴러가는 데 필요한 일이면 직접 메운다
- 직접 담당: QA 테스트 · 공지/이벤트 웹뷰 · 이메일 CS · 콘텐츠 검열 등 운영 전반

**그래서 남들이 못 보는 진짜 병목을 안다**
- 직접 부딪힌 페인포인트를 곧장 도구·자동화로 전환 → **운영 자동화 9건**
- 예: 이미지 자동 검열 · 이벤트 보상 지급 등을 이벤트 기반 Lambda로 셀프서비스화
- "직접 운영했기에 진짜 병목을 알았다" — 책상 위 추측이 아니라 현장에서 나온 우선순위

## Slide 4 — ③ 빠른 러닝커브 (AI 레버리지)

> 특정 스택에 묶이지 않는다 — AI를 지렛대로 새 기술도 빠르게 적응해 *제품까지* 만든다.

**새 도메인을 직접 제품으로 증명**
- **프론트엔드(React) 미경험** → AI 코딩 도구로 맛보기 채팅 웹 직접 구현·배포
- **이미지 생성(Stable Diffusion) 새 도메인** → 이미지 생성 인프라 0→1(일 3,000건·5초) + ComfyUI API 서버 직접 제작·공개

**왜 가능한가**
- "읽고 넘기지 않고 바로 적용" — 새 방식이 보이면 지금 프로젝트에 즉시 적용해 스킬로 굳힌다
- AI 코딩 시대의 병목은 "어떻게 짜지"가 아니라 "무엇을 원하고 결과를 판별하나" → 스택 장벽이 낮아짐
- 결론: **기술스택은 제약이 아니다. 필요한 걸 빠르게 익혀 제품으로 만든다.**

## Slide 5 — ④ 시행착오라는 자산

> 진짜 자산은 '있었던 것'이 아니라 '깨진 것'. 비싸게 배웠기에 같은 실수를 반복하지 않는다.

**완벽주의 → 최속 검증**
- 깨짐: 완벽히 다듬어 출시했더니 아무도 안 원함 / 잡은 수요는 기다리다 이탈
- 재발 방지: 오버엔지니어링 경계 + MVP 규율(좋은 아이디어도 의도적으로 버림) → RAG 폐기·단순화

**기능 집착(solution-first) → 문제부터 보기(problem-first)**
- 깨짐: 지표 하락에 데이터 진단 없이 "기능 부족"이라 단정, 기능만 추가
- 재발 방지: 그로스해킹을 '제2의 무기'로 → Statsig A/B 인프라 구축, 기능 전 "어떤 문제? 무엇으로 검증?"

**기술적 한계의 정직한 회고**
- 깨짐: 정밀의료 AI 문진 — 데이터 증강·모델 업그레이드로도 domain adaptation 못 풀음
- 배움: 데이터·모델 크기만으론 도메인 특화 성능 담보 불가 → 별도 적응 전략 필요
