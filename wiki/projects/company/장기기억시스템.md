# 장기기억 시스템

**소속**: [[entities/companies/레플리\|Reppley]]

## 문제

캐릭터와 오래 대화할수록 이전 대화가 반영되지 않아 사용자가 "나를 기억하지 못한다"고 느끼는 문제가 있었다.

기억의 연속성이 끊기면 캐릭터와의 **유대감과 몰입도**가 떨어졌고, 이는 서비스 만족도에도 직접적인 영향을 주는 핵심 과제였다.

## 업계 현황 분석

당시 장기기억 시스템의 업계 표준은 **Hybrid RAG(BM25 + Vector Search)** 기반 구조였다. AWS 세션과 경쟁사 사례를 조사했을 때도 대부분 유사한 접근을 사용하고 있었다.

한편, 사용자와 장기간 상호작용하는 AI 서비스에서는 **Episodic Memory, State Memory, Memory Graph** 등 인간의 기억 구조를 참고한 새로운 접근법들이 등장하고 있었다.

나는 캐릭터 챗 서비스의 핵심 문제를 단순 정보 검색이 아닌 **"관계와 경험의 지속성"**이라고 보았다. 사용자가 원하는 것은 과거 대화를 정확히 검색하는 것보다, 캐릭터가 이전 경험과 관계 변화를 일관되게 기억하는 것이었다.

## 내부 설득

업계 표준인 Hybrid RAG를 먼저 프로덕션 후보로 검토했다.

그러나 현재 서비스 규모와 문제 특성을 고려했을 때, 검색 계층보다 먼저 **기억 구조 자체를 설계하는 것**이 더 중요한 과제라고 판단했다.

이를 검증하기 위해 아래 내용을 정리한 발표 자료를 작성해 대표님과 팀원들에게 공유했다:

- Hybrid RAG 도입 시 예상 비용 구조
- 운영 복잡성 증가 요인
- 임베딩 모델 의존성과 유지보수 비용
- 캐릭터 챗 도메인에 적합한 기억 구조
- Episodic / State Memory 기반 접근 사례 조사

검토 결과, 먼저 Episodic + State 기반 기억 시스템을 구축하고 Retrieval은 이후 필요 시 확장하는 방향으로 합의했다.

이는 Retrieval을 부정한 것이 아니라, **문제의 우선순위를 고려해 단계적으로 접근한 결정**이었다.

## 구현

- **경량 LLM + Redis 조합** — Vector DB 없이 시스템 단순화
- **LLM-as-judge** — 컨텍스트 추출 정확도 자체 평가 파이프라인 구축
- **자체 벤치마크** — 장기기억 품질 점수 설정 → 점수가 오르는 방향으로 반복 튜닝

## 스택

`AWS ElastiCache (Redis)` · `경량 LLM`

## 결과

- 사용자 기억력 체감 품질 향상 (자체 벤치마크 기준)
- 운영 비용 감소 (RAG 대비 비용·인프라 복잡성 절감)
- MAU 3만 · 일 70만 건 규모에서 안정 운영

## 의의

이 프로젝트에서 가장 중요한 의사결정은 "최신 기술을 도입할 것인가"보다 **"지금 가장 중요한 문제를 무엇으로 정의할 것인가"**였다.

> "기술 선택은 정답을 찾는 과정이 아니라, 현재 문제에 가장 적합한 트레이드오프를 선택하는 과정이다."
