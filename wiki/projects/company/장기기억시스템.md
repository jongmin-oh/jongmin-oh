# 장기기억 시스템

**소속**: [[entities/reppley\|Reppley]]

## 문제

오래 대화할수록 캐릭터가 이전 대화를 기억하지 못해 **유대·몰입이 끊긴다**는 사용자 불만 누적.

## 의사결정 — RAG 검토 후 폐기

업계 표준인 RAG(Retrieval-Augmented Generation)를 먼저 검토했으나 4가지 이유로 폐기:

| 이유 | 내용 |
|------|------|
| 운영 비용 | MAU 3만 · 일 70만 건에서 Vector DB·임베딩 호출 단가 부담 |
| 운영 복잡성 | 컴포넌트 추가로 인프라·운영 부담 증가 |
| 아키텍처 락인 | 임베딩 모델 사실상 영구 고정, 평가에도 비용, 변경 시 전체 재임베딩 필요 |
| 도메인 가설 | 캐릭터 챗봇의 "사람 같은 기억"은 LLM 자체 능력으로 구현 가능 |

→ **더 단순한 가설(LLM + Redis)부터 검증**하기로 결정.

## 구현

- **경량 LLM + Redis 조합** — Vector DB 없이 시스템 단순화
- **LLM-as-judge** — 컨텍스트 추출 정확도 자체 평가 파이프라인 구축
- **자체 벤치마크** — 장기기억 품질 점수 설정 → 점수가 오르는 방향으로 반복 튜닝

## 스택

`AWS ElastiCache (Redis)` · `경량 LLM`

## 결과

- 사용자 기억력 체감 품질 향상 (자체 벤치마크 기준)
- 운영 비용 감소 (RAG 대비 비용·인프라 복잡성 절감)
- MAU 3만 · 일 70만 건 규모에서 안정 운영

## 의의

이 프로젝트는 [[skills/working-style\|오버엔지니어링 경계]] 철학의 가장 선명한 사례:
> "업계 표준이 있더라도, 더 단순한 가설이 먼저다."

RAG 폐기 의사결정의 4가지 이유는 나중에 다른 기술 선택에서도 재사용할 수 있는 프레임워크.
