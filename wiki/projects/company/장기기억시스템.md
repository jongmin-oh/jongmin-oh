# 장기기억 시스템

**소속**: [[entities/companies/레플리\|Reppley]]

## 문제

오래 대화할수록 캐릭터가 이전 대화를 기억하지 못해 **유대·몰입이 끊긴다**는 사용자 불만 누적.

## 업계 현황 분석

당시 챗봇 장기기억의 업계 표준은 **하이브리드 RAG** — 키워드 검색(BM25) + 시맨틱 서치(임베딩)를 결합하는 방식이었다. AWS 세션에서 경쟁사 발표를 직접 들었을 때도 이 전략을 쓰고 있었다.

한편 SuperMemory 등 새로운 접근법이 등장하고 있었다: **Episodic Memory** — 대화를 벡터로 인덱싱하는 대신, 사람의 기억처럼 에피소드 단위로 저장하고 State를 기반으로 유지하는 방식.

종민의 가설: **"사람의 기억은 키워드 검색이 아니라 에피소드와 상태(State) 기반으로 작동한다. 캐릭터 챗봇의 '사람 같은 기억'도 그 구조가 더 맞다."**

## 내부 설득 — 발표 자료 준비

업계 표준(하이브리드 RAG)이 강력한 레퍼런스로 작용해 대표님과 팀원들을 설득하기 쉽지 않았다. 직접 **발표 자료를 준비**해 아래를 논거로 제시했다:

- 하이브리드 RAG의 비용·복잡성·락인 문제
- Episodic/State 기반 기억이 챗봇 도메인에 더 적합한 이유
- SuperMemory 등 신규 방법론 동향

설득에 성공해 에피소드·State 기반 접근으로 구현했다.

## 의사결정 — RAG 검토 후 폐기

업계 표준인 RAG(Retrieval-Augmented Generation)를 먼저 검토했으나 4가지 이유로 폐기:

| 이유 | 내용 |
|------|------|
| 운영 비용 | MAU 3만 · 일 70만 건에서 Vector DB·임베딩 호출 단가 부담 |
| 운영 복잡성 | 컴포넌트 추가로 인프라·운영 부담 증가 |
| 아키텍처 락인 | 임베딩 모델 사실상 영구 고정, 평가에도 비용, 변경 시 전체 재임베딩 필요 |
| 도메인 가설 | 캐릭터 챗봇의 "사람 같은 기억"은 Episodic/State 기반이 더 적합 |

→ **경량 LLM + Redis로 Episodic 기억 구조 구현**하기로 결정.

## 구현

- **경량 LLM + Redis 조합** — Vector DB 없이 시스템 단순화
- **LLM-as-judge** — 컨텍스트 추출 정확도 자체 평가 파이프라인 구축
- **자체 벤치마크** — 장기기억 품질 점수 설정 → 점수가 오르는 방향으로 반복 튜닝

## 스택

`AWS ElastiCache (Redis)` · `경량 LLM`

## 결과

- 사용자 기억력 체감 품질 향상 (자체 벤치마크 기준)
- 운영 비용 감소 (RAG 대비 비용·인프라 복잡성 절감)
- MAU 3만 · 일 70만 건 규모에서 안정 운영

## 의의

> "업계 표준이 있더라도, 더 단순한 가설이 먼저다."

RAG 폐기 의사결정의 4가지 이유는 나중에 다른 기술 선택에서도 재사용할 수 있는 프레임워크.
