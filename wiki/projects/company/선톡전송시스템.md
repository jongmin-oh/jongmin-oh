# 선톡 전송 시스템

**소속**: [[entities/companies/레플리\|Reppley]]

## 문제

사용자가 챗봇과 대화 후 시간이 지나면 자연스럽게 이탈. "내가 말 걸 때만 응답하는 관계"로는 유대·몰입에 한계. → **캐릭터가 먼저 말 걸어오는 경험**으로 리텐션 향상.

## 기능

| 타입 | 설명 |
|------|------|
| Continue | 이전 대화 맥락을 이어가는 선톡 |
| WakeUp | 새로운 대화를 시작하는 선톡 |

## 아키텍처 결정

**메인 서버에 붙이지 않고 별도 Lambda로 분리.**  
이유: 메인 서버에 추가하면 복잡성 증가. 기존 백엔드 파이프라인(Lambda Step Function 기반 선톡 대상 추출)에 자연스럽게 Lambda를 연결하는 것이 더 깔끔하다고 판단.

```
Lambda Step Function (선톡 대상 추출)
  → SQS (대상 채팅방 목록)
  → Lambda (SAM) — LLM 호출 · 메시지 생성
  → 기존 채팅 발송 시스템 재사용
```

## 한 일

- **시스템 분리** — 별도 Lambda(SAM) + SQS 파이프라인 구축
- **메시지 생성** — 채팅방별 LLM 호출, 캐릭터 페르소나 반영한 선톡 생성
- **대량 처리** — SQS로 LLM 호출·발송 비동기 처리, 동시 호출 폭주 방지
- **프롬프트 개선** — 7일간 실제 생성 결과 **약 5만 건을 직접 검토**하며 페르소나 일관성·자연스러움 기준으로 반복 개선

## 스택

`AWS Lambda (SAM)` · `AWS SQS` · `Python`

## 결과

- 일평균 **8,000건** 발송 안정 운영
