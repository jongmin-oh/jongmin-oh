# 운영 도구 내재화

**소속**: [[entities/companies/레플리\|Reppley]]  
**성격**: 자발적 프로젝트 · 본인 업무 외 페인포인트까지 직접 도구화

## 핵심 맥락

> 운영을 남 일로 두지 않고 직접 겪었기에 진짜 병목을 알았고, 그걸 일회성 처리가 아니라 시스템으로 바꿨다.

QA·공지사항·이벤트 페이지·이메일 CS·콘텐츠 검열 등 **운영 전반을 직접 담당**하며 반복·SQL 직접 처리·수작업 페인포인트가 누적 → 직접 도구화.

팀 내 **전담 운영 담당자가 없었기 때문에** 본인이 운영까지 맡아야 했고, 빠른 처리를 위해 **모바일에서도 접근 가능한 Slack 인터랙션**을 팀 내 보편 운영 채널로 구축·정착시켰다.

## 관리자 페이지 (셀프서비스 전환) — 4종

| 도구 | Before | After |
|------|--------|-------|
| 큐레이션 관리자 | 메인 노출 챗봇 변경마다 개발자 SQL 직접 처리 | 담당자 드래그앤드롭 |
| 공지사항 관리자 | 텍스트·이미지 S3 업로드 + SQL | 담당자 직접 작성·발행 |
| 테스트 어드민 | 재화 0·출석 초기화 등 QA 반복 요청 → 개발자 처리 | 셀프 처리 |
| 운영 어드민 | 하트 지급·문의 답변·커스텀 푸시·유저 조회 SQL/DynamoDB | UI 통합 |

## Slack 인터랙션 운영 채널 — 실시간 모바일 대응

팀 전체가 사용하는 Slack 인터랙션을 구축해 별도 어드민 페이지 없이도 즉시 처리 가능한 운영 흐름을 만들었다. 모바일에서도 접근 가능해 자리를 비운 상태에서도 빠른 대응이 가능했다.

| Slack 인터랙션 | 처리 내용 |
|---------------|----------|
| 이벤트 하트 지급 | 인터랙션 버튼으로 유저에게 하트 즉시 지급 |
| 사용자 재재 (정지/제한) | 문제 유저 계정 즉시 처리 |
| 콘텐츠 재재 | 불건전 게시물·챗봇 즉시 비활성화 |
| CS 알림 | 유저 문의 이메일 → Slack 자동 라우팅 + 인터랙션 응답 |
| 불건전 챗봇 필터링 | 자동 감지 후 Slack 알림 → 버튼 한 번으로 처리 |

## 자동화 — 5종

| 자동화 | Before → After |
|--------|---------------|
| SNS 이벤트 보상 | 구글폼 → 스프레드시트 → Postman 수동 지급 → 자동 지급 |
| 앱 테스트 | 출시 전 수작업 기본 테스트 → Appium + Small TC 자동화 |
| 캐릭터 이미지 자동 검열 | 부적절 사진이 실시간 랭킹에 1일 이상 노출된 사건 이후 자동 검열 구축 |
| 채팅 데이터 로그 수집·분석 | 사용량 질문 응대 → 슬랙 정기 리포트 |
| 유저 문의 이메일 → 슬랙 | 메일 본문 매번 복붙 → 자동 라우팅 |

## 스택

`FastAPI` · `JavaScript` · `HTML` · `CSS` · `AI 코딩 도구 (Claude Code 등)` · `Appium` · `Slack API (Block Kit · Interactivity)`

## 결과

- 개발자 손이 가던 **9개 반복 작업** 운영팀·QA·담당자 셀프서비스로 전환
- 이미지 노출 사고 재발 방지 시스템 구축 (이미지 자동 검열)

## 의의

[[skills/working-style\|"업무의 경계를 가리지 않는다"]] 철학의 집약. 개발자가 운영 전반을 직접 경험했기 때문에 **진짜 병목**을 파악했고, 그게 9개 도구·자동화로 이어졌다.

특히 Slack 인터랙션을 팀 내 운영 인프라로 정착시킨 것은 단순 자동화를 넘어 **"운영 담당자 없이도 돌아가는 운영 체계"** 를 만든 사례다 — 자리를 벗어난 상태에서도 모바일로 즉각 대응 가능했다는 점에서 운영 속도와 리스크 대응 모두 해결했다.
