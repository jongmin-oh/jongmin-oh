# 일하는 방식 & 엔지니어링 철학

## 핵심 원칙 7가지

### 1. 업무의 경계를 가리지 않는다

개발에 그치지 않고 **제품이 굴러가는 데 필요한 일이면 직접 메운다**.

증거: QA 테스트 전담, 공지사항 관리, 이벤트 웹뷰 관리, 이메일 CS 대응, 콘텐츠 검열 등 운영 전반 직접 담당 → 그렇게 직접 부딪힌 페인포인트를 도구·자동화로 해결한 것이 [[projects/company/운영도구내재화\|운영 도구 내재화 9건]].

> 직접 운영을 담당했기 때문에 **진짜 병목**을 알았다.

---

### 2. 오버엔지니어링 경계

간결함 최우선. **"정말 필요한가"** 를 먼저 묻고, 가장 단순한 가설부터 검증.

증거: [[projects/company/장기기억시스템\|장기기억에서 RAG를 검토 후 폐기]] → LLM+Redis로 단순화.  
폐기 이유 4가지: 운영 비용 / 운영 복잡성 / 아키텍처 락인 / 도메인 가설.

---

### 3. 서버리스 우선 — 신경 덜 쓰고 최속 검증

가능하면 서버리스로 설계. 운영 부담과 고정 비용 최소화, 트래픽 탄력성 확보.

**목적은 "최소한의 운영 신경으로 가장 빠른 검증"이다.** 인프라·서버 관리 같은 곁가지에 인지 자원을 뺏기지 않고, 아키텍처를 의도적으로 단순하게 가져가 **초기 제품(POC)을 가장 빠르게 시장에 올려 가설을 검증**하는 데 집중한다. → 이것이 본인의 헤드라인 강점이다 (→ [[overview]]).

이 강점은 공짜로 얻은 게 아니다. [[reflections/초기-스타트업-시행착오]]의 클러스터 A("완벽주의가 타이밍을 죽인다")에서 비싸게 배운 교훈 — *빨리 시장에 내놓고 검증하라* — 이 능력으로 굳은 것이다. 교훈 ↔ 강점은 같은 동전의 양면.

**AWS Lambda + SAM에 매우 익숙**, 빠르게 구현·배포.

| 적용 사례 | 방식 |
|---------|------|
| [[projects/company/선톡전송시스템\|선톡 시스템]] | Lambda(SAM) + SQS |
| 이미지 자동 검열·이벤트 보상 등 | 이벤트 기반 Lambda |
| [[projects/company/맛보기채팅웹\|맛보기 웹]] · [[projects/personal/널스체크\|널스체크]] | S3 + CloudFront |

단, GPU 추론처럼 서버리스가 불리한 워크로드는 직접 GPU 계약으로 전환 → **워크로드 특성에 맞게 선택**.

---

### 4. 비용·복잡성 우선 고려

기능을 넣기 전에:
- 운영 비용이 선형으로 늘지 않는지?
- 운영 부담이 커지지 않는지?

증거: 대화 요약 + 프롬프트 캐싱(건당 비용 30% 절감) / 번역 키워드 캐싱(선형 증가 차단) / GPU 직접 계약(SaaS 대비 5배 절감).

---

### 5. 트레이드오프 직접 진단

기존 도구의 한계가 명확하면 **직접 만들어 해결**.

증거: [[projects/personal/ComfyUI-API-서버\|ComfyUI API Server]] — ComfyUI(빠르지만 API 없음) ↔ SD WebUI(API 좋지만 느림) 트레이드오프를 동시 해결하는 헤드리스 서버 직접 구현.

---

### 6. 명세 주도 개발 (SDD)

AI가 코드를 써주는 시대에 병목은 **"어떻게 짜지"가 아니라 "무엇을 원하는지 정의하고 결과를 판별하기"** 로 옮겨갔다.

SDD 흐름:
1. 명세 작성 (추상 → 구체·수치화)
2. 애매한 갈림길을 질문으로 합의 (grill-me)
3. 검증 가능한 성공 기준 수립
4. 새 파일에서 위험 격리 후 구현
5. 실제 데이터로 목표에 닿을 때까지 한 번에 하나씩 바꿔 측정
6. 트레이드오프·한계 정직하게 정리

### 7. 24시간 실시간 B2C 운영 경험

**멈추면 안 되는 서비스를 직접 굴려본 감각.** 캐릭터 챗봇은 정해진 업무 시간이 없는 B2C 서비스다 — 새벽 3시에도 일 최고 70만 건의 대화가 돌아가고, 장애·지연·부적절 응답은 곧바로 사용자 이탈로 이어진다.

이 경험에서 체화된 것:
- **실시간 트래픽 감각** — 무중단 운영, 트래픽 스파이크 대응, 24/7 모니터링·알림을 전제로 설계한다.
- **사용자 반응의 즉시성** — 배포 직후 곧바로 실사용자 반응이 돌아온다. 지표 변화와 이상을 빠르게 읽고 대응하는 루프가 몸에 배어 있다.
- **악용·콘텐츠 리스크의 상시성** — 사람이 자는 시간에도 불건전 사용·악용이 들어오므로, 자동 검열·필터링·알림을 운영의 기본값으로 깐다 (→ [[projects/company/운영도구내재화]]).

> 이건 **대부분의 B2B 개발자가 경험하기 어려운 강점**이다. B2B는 보통 정해진 클라이언트·SLA·업무 시간 안에서 움직이지만, 24시간 익명 대중을 상대하는 B2C 운영은 트래픽 패턴, 장애 영향, 사용자 행동의 결이 근본적으로 다르다. 본인은 이 차이를 직접 겪어서 안다.

원칙 1(경계 없는 운영)이 *무슨 일이든 직접 메운다*는 폭이라면, 이 원칙은 *멈추지 않는 B2C를 24시간 책임져봤다*는 깊이·결의 차이다.

## 학습 태도

**읽고 넘기지 않고 바로 적용.**

흘려들은 지식은 끝내 내 것이 되지 않는다고 본다. 새로운 방식이 보이면 지금 하는 프로젝트에 직접 적용해 스킬로 굳힌다.

증거: SDD를 한 프로젝트에 몰아 적용해 체화. React 미경험 → AI 코딩 도구로 맛보기 웹 직접 구현.
